Standing rule for this page

Mocked/unit tests don't count as coverage anywhere here — every number, every cell, every "passing" is a real test against a real LLM/CLI, never a mock. Total test-file counts appear once, for scale, then are immediately narrowed to the real-e2e subset; a unit test proving construction logic in isolation says nothing about whether the real thing works, and isn't treated as if it does.

Layer 0 · third party

The CLI binaries

Layer 1 · adapter · the rigorous layer

multi-llm-provider-go

Its whole job: make each individual provider correct on its own terms — no cross-provider abstraction, that's Layer 2's. The only layer with a formal priority system: CodingAgentCertificationPriority. P0 = release-blocking, one gate (-coding-cli-p0-live) — a provider isn't usable if any P0 proof is missing. P1 = everything else, real coverage, not release-blocking.

tmux — the 13 P0 certifications

requiredP0CertificationIDs (12) plus structured_streaming, which the code marks P0 "wherever it is required" — currently that's all four providers, so 13 in practice today:

  • fresh_launch — starts in the right workspace
  • runtime_context — system prompt/skills/MCP runtime reach the provider
  • working_directory — provider operates in the exact caller workspace
  • trust_auth_prompts — startup trust/auth gates clear before the first prompt
  • mcp_bridge — bridge launches and tools are callable
  • slow_tool_false_idle — a long tool call is never misread as "done"
  • done_detection — the adapter knows when a turn is really finished
  • final_extraction — only the real answer is returned, never chrome/tool dumps
  • live_input / busy_live_input — steering works, including while busy
  • cancellation — context cancellation actually stops the process
  • parallel_isolation — concurrent sessions don't cross-talk
  • structured_streaming — clean text/tool-call extraction during a live tmux turn (not the JSON transport below — see the callout in the companion doc)

Structured/JSON transport — real tests, no registry

The gap in this layer

Real, passing tests for 3 of 4 providers — but only one (TestPiCLIStructuredStreamingRealWorldLive) is wired into CodingAgentProviderCertifications, and it proves the unrelated structured_streaming cert, not this transport. Nothing stops a structured-transport regression from shipping silently the way a tmux one can't.

The 7-item conceptual bar — same P0 label as the tmux registry above, just not wired into it:

  • fresh_launch_structured — real CLI launch in structured mode, real text response
  • token_usage_structured — result/finish events produce non-zero, correctly-attributed token counts
  • tool_call_streaming_structured — tool calls stream as distinct start/end chunks, not buried in text
  • mcp_bridge_structured — a real MCP bridge tool is callable under this transport too
  • bridge_only_structured — built-in tools can be denied while bridge tools still work
  • error_handling_structured — a clear error surfaces on process failure or empty output, never a silent hang
  • graceful_shutdown_structured — the documented SIGTERM×3→SIGKILL sequence fires on the terminal event

Layer 2 · library · the thin layer

mcpagent

Its whole job is the mirror of Layer 1's: mask which provider and transport is underneath, one consistent interface on top — system prompt (with resume), MCP bridge, skills, conversation persistence, cost accounting, transport selection. Most of that is real and built (system prompt, bridge, skills, multi-turn, steering); transport selection and conversation/cost ownership are still design targets, not shipped code (see companion doc §Four layers, four owners).

No formal registry exists here — "P0" appears only 4x in the whole tree, all informal comment labels, not a machine-checkable priority. Narrower gap than "owns nothing" though: the responsibilities themselves mostly do have real e2e proof, just not organized into a P0/P1 system.

Steering on json was mis-wired — found and fixed this pass

The verification pass for this matrix turned up a real defect, not just a hole. Agent.SupportsSteering() (coding_session.go) and DeliverUserMessage() (message_delivery.go) both decided "can I inject live input into the running turn?" purely from contract.SupportsLiveInput — a provider+model property — and never consulted the per-agent CodexStructuredTransport/CursorStructuredTransport/PiStructuredTransport flags. So on a structured/json run SupportsSteering() still returned true, and Deliver() on a busy session routed to steer → tried to push tmux live-input into a one-shot codex exec --json / cursor-agent --print / pi --print --mode json process with no live pane. The original author flagged exactly this ("future query-only transports … will not") but never wired the check. Fixed: a new usesStructuredTransport() predicate (union of the per-provider flags, ForceStructuredCodingAgent, and a non-tmux native contract) now gates both entry points, so a busy structured turn routes to decideQueue. Proven deterministically (TestSupportsSteeringFalseOnStructuredTransport, TestDeliverQueuesOnStructuredCodingAgent — all providers × all three structured-activation paths, tmux controls intact) and live against all three real structured CLIs (TestStructuredTransportDeliverQueuesMidTurn: Deliver returns DeliveryModeQueued, and the queued message is provably injected at the next LLM-call boundary and obeyed — Codex/Cursor/Pi all green, agent-reviewed). Found live: on a query-only transport the steer queue drains at the next LLM-call boundary, which is often within the same ContinueConversation, not a separate next turn.

Native --resume on json — investigated deep, then implemented across all 4 providers

Proving multi-turn on json surfaced that native --resume (unlike history-replay, which already worked 3/3 via convrecord) was broken or unwired for every provider, each differently — a layered investigation, not one bug. (1) mcpagent: the continuation handle recorded transport=tmux for structured turns (same Force-vs-per-provider-flag blind spot as the steering bug) — fixed with the shared usesStructuredTransport() predicate. (2) Codex: codex exec resume <id> was deliberately unwired (a distinct subcommand that drops --profile/--sandbox/-C) — implemented by layering the MCP profile via the GLOBAL --profile flag (bridge secrets stay in the file, never argv), sandbox via a global -c, cwd via the process dir. (3) Cursor: --resume was wired but never proven; a write-based test conflated it with cursor's --mode ask write-refusal — a recall-based turn 2 isolated continuity and proved it. (4) Pi: structured neither surfaced its session id nor passed a session flag — Pi's earlier apparent pass was the model re-reading the file, which the recall test exposed; wired --session-id capture-and-resume. All three now recall turn 1's build id purely from restored session context (recall forbids re-reading), agent-reviewed; tmux native-resume regression still 4/4.

Claude's json lane — built this session (was the last n/a)

Claude previously had no one-shot json lane, so every Claude json cell was n/a and "3 providers" was the ceiling. That's closed: a new claudecode_structured_adapter.go drives claude -p --output-format stream-json (mirroring the codex/cursor/pi structured adapters), wired end to end through mcpagent (WithClaudeCodeStructuredTransport → usesStructuredTransport() → integration options). Native --resume works (claude --session-id/--resume; the id was already captured via claude_code_session_id). Bridge-only needed a real fix: --allowedTools is only a permission whitelist that --dangerously-skip-permissions overrides (found live — claude fell back to native Bash when the bridge failed), so the adapter adds a hard --disallowedTools denylist. All 7 structured e2e tests pass for Claude, agent-reviewed. json is now 4 providers, not 3, for every shared capability.

Two real Pi bugs found and fixed getting multi-turn/continuity/steering to 4-provider coverage

1. MCP_READY_FILE — a fresh random marker path generated on every call by design — was leaking into the persistent-session reuse fingerprint, so every turn computed a different hash than the registered session and a perfectly good, still-alive tmux session was discarded and relaunched from scratch every turn. 2. Once reuse started working, waitForPiMarkerType/waitForPiPromptReady (waiting for the "session_start" marker) ran unconditionally even on a reused session — but that marker only fires once per actual process launch, so turn 2 hung forever waiting for an event that would never come again. Both confirmed via live debug instrumentation, fixed by gating on createdSession (matching the pattern the adjacent MCP-ready-file wait already used), and verified with real passing runs.

Two real Cursor bugs found and fixed getting tool-failure/multi-turn to 4-provider coverage

1. newRealBridgeAgentWithShell (used by the tool-failure tests) was missing WithSessionID(...) — present in every other test's agent-construction helper. Without it, Cursor's captured answer included the full verbatim input prompt (prompt-echo), and this test's own Go tool-handler counter never incremented. Fixed by adding the session ID; the prompt-echo disappeared completely. The counter still read 0 afterward — traced to Cursor routing tool calls through its own GetMcpTools/CallMcpTool meta-tool wrapper rather than the directly-registered handler other providers hit; the test's assertion was widened to also accept a real streamed tool-call event as proof of attempt, which is the correct provider-agnostic signal. 2. freshestCursorStoreDBSince (the transcript-tailing poller behind structured streaming) only checked store.db's own mtime to decide if a candidate file was fresh enough for the current turn. SQLite writes in WAL mode land in store.db-wal without ever touching store.db's mtime until a checkpoint — so on a reused, persistent session, turn 2's store.db (same file as turn 1's) permanently read as "too old" and the poller silently found nothing, producing zero streaming events every turn after the first. Fixed by judging freshness from the newest mtime across store.db/-wal/-shm, while still reading the store.db path itself (the read-only connection sees committed WAL contents transparently). Both confirmed via live debug instrumentation (candidate-mtime logging), fixed, and verified with real passing 4/4 runs.

Reading the scorecard straight

Where the rigor actually is, ranked

  • 1. Layer 1, tmux — real, enforced, release-blocking.
  • 2. Layer 1, structured — real, passing — but not enforced. A regression ships silently until someone runs the test manually.
  • 3. Layer 2 · tmux — all core capabilities now proven across all 4 providers (not just Claude): bridge, tool-failure, multi-turn, concurrency, steering, continuity, system-prompt, skills, convrecord. Only Pi's documented message-modes non-bug remains a "3/4". Not organized into a P0/P1 registry.
  • 4. Layer 2 · json — was the build-out front, now closed. Proven at Layer 2 across all 4 providers: system-prompt, skills, convrecord, steering-queues, tool-failure, and multi-turn via native --resume — which required real cross-CLI adapter work (Codex exec resume, Pi --session-id, a whole new Claude structured adapter) plus two transport-blindness fixes. bridge/streaming/concurrency scoped out as covered-elsewhere/low-risk. Not yet organized into a P0/P1 registry — and its P0 set differs from tmux's (queue-not-steer, native --resume, one-shot lifecycle).
  • 5. Layer 3 — unknown, not measured.

Reusable knowledge · read before touching transport code

Findings & learnings that will save the next person hours

Hard-won during the tmux+json coverage push. Each of these was a wrong assumption that cost real debugging time — captured here with the concrete file/flag names so the next change starts from the answer, not the symptom.

1 · The transport-blindness bug class (bit twice — will bite again)

Rule: transport decisions must use usesStructuredTransport(), never just ForceStructuredCodingAgent

A coding agent runs structured/json via either ForceStructuredCodingAgent (the workflow transport="structured" switch) or a per-provider flag (CodexStructuredTransport/CursorStructuredTransport/PiStructuredTransport). Any code that checks only the Force flag is transport-blind and silently mishandles per-provider-flag sessions. This exact blind spot caused two separate production-relevant bugs: (a) SupportsSteering()/DeliverUserMessage() tried to tmux-inject into a one-shot json process instead of queuing; (b) the continuation handle (legacyCodingProviderSessionHandle) labelled structured turns transport=tmux, so turn 2 resumed through the wrong path and lost all context. Both fixed with the shared Agent.usesStructuredTransport() predicate (union of the Force flag + the three per-provider flags + a non-tmux native contract). Grep for ForceStructuredCodingAgent before adding any transport branch.

2 · Native --resume on json — the per-provider reference

Every structured provider resumes differently, and none of it is a flag on the fresh command. The two-sided contract: turn 1 must surface its native session id in GenerationInfo.Additional (captured by extractCodingAgentSessionIDs → a.<Provider>SessionID); turn 2 must consume it. 3 · Test-design learnings (the tests themselves were wrong before the code was)

  • Continuity tests must RECALL, never WRITE. A turn-2 that writes a file to prove it remembered turn 1 conflates continuity with write-capability (Cursor's structured --mode ask refuses natural-language writes) and lets a model cheat by silently re-reading the source file. A turn-2 that forbids file-read/tool-use and asks the model to recall the value from context is the only rigorous proof. This is what exposed Pi's fake pass — its write-turn "worked" only because it re-read the file; forbidding that, it hallucinated, revealing native resume was unwired.
  • A canary word that can only exist via the real path is self-validating — system-prompt/skill survival tests need no LLM-judge because the secret word cannot appear unless the prompt/skill genuinely reached the model.
  • agentreview fingerprints key off the shape, not the content — a reviewed verdict survives reruns with different random build-ids as long as the structural shape is unchanged, so you approve once per shape, not per run. Use _ separators in test-name args (a / creates an unwritten nested dir the flat-scan gate can't see).
  • Stream deltas concatenate verbatim; only complete blocks join with \n. captureRealBridge inserting a newline after every chunk split build-ids mid-string (BUILD_ID_4aa\n94683070c). Delta-aware joining (mirror buildMessageModes) is mandatory for any provider that streams fine-grained tokens (Pi).

4 · Provider quirks that look like bugs but aren't

  • Codex's native functions.exec is unremovable by any flag. It bypasses a failing bridge tool and reads files directly, so "bridge-only" and tool-failure "give-up" scenarios are unfalsifiable for Codex — gate with strictBridgeOnly=false / skip give-up for Codex. Recovery still tests fine.
  • Pi narrates using raw tool/API names (api_bridge_execute_shell_command) in its own prose — a message-modes banned-substring check correctly flags it. It's real model verbosity, not a chrome leak; leave the check strict rather than weaken it for everyone.
  • Cursor structured store.db uses SQLite WAL — store.db's own mtime never advances until a checkpoint, so a freshness check on it alone reads a reused (turn-2) db as stale and the transcript poller finds nothing. Judge freshness across store.db/-wal/-shm.
  • Cursor's first tool-call can take >5 min under load — the steer/multi-turn e2es are opt-in (RUN_MCPAGENT_STEER_E2E) precisely because of this live-model timing flakiness; a single miss is not a regression.
  • The query-only steer queue drains at the next LLM-call boundary, which is usually within the same ContinueConversation (after a tool result), not a separate next turn — so a "queued" message can legitimately land in the same turn's answer.

5 · Disk & session lifecycle

  • Persistent tmux sessions intentionally survive Agent.Close() (so a later turn reuses them) — but a test process never returns, so persistent-session e2es leak their tmux session unless they explicitly call Close<Provider>CLIInteractiveSessionForOwner. 134 orphaned mlp-* sessions accumulated before this was added; enough clutter to hang the concurrent test via tmux-server strain.
  • Bounded (non-persistent) sessions rely on a 30s in-process delayed kill that never fires during an isolated -run (the process exits first) — a known minor leak; sweep manually between isolated runs.
  • Projected skills + prompt leak into a REAL workdir on close. Isolated workspaces are rm -rf'd wholesale; a real workdir keeps the active provider's artifacts. Skill subdirs: Claude .claude/skills, Cursor .cursor/skills, Codex .agents/skills (NOT .codex — its config teardown misses skills), Pi .pi/skills. Prompt files: CLAUDE.md/AGENTS.md (marker mlp-session-instructions), Pi .pi/APPEND_SYSTEM.md (NOT GEMINI.md). Agent.Close() now removes exactly the named skill folders + marker-verified prompts — operator content untouched.
  • A bridge-only agent with no file-read tool cannot open a projected SKILL.md — deny-builtin providers (Claude/Cursor/Pi) have no way in; only Codex's native exec could. A realistic agent always registers the bridge shell tool, so test builders must too.

The meta-learning

tmux and json are not the same tests mirrored — the json P0 set is genuinely different: steering means queue (never steer), multi-turn means native --resume (not persistent-pane reuse), lifecycle is one-shot. Every place that reasons about "the coding agent" must ask which transport first. And the recurring shape of the real bugs here was the same: a decision that keyed off ForceStructuredCodingAgent or a provider contract when it should have keyed off the actual transport of this session.

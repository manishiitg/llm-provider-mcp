tmux vs. JSON streaming, cheaper-but-harder-to-automate — that's the correct shape of the trade-off. What's missing is: there are three channels, not two; steering has no equivalent in a one-shot channel; and some of JSON's advantage may already be won, on the current transport, by the message-reconstruction work already built.

Add to the map

  • Direct API is a third lane, not a variant of streaming — no subscription pricing, no TUI noise, no session/tmux lifecycle at all, but no flat-rate cost ceiling either.
  • Live steering (mid-turn) only exists where a process is still running. A one-shot streaming or API call has nothing to inject into — tmux's only truly exclusive capability, and it's one both apps now use.
  • The clean tool/text extraction "streaming" promises is partly already delivered on tmux, via the transcript/marker parsing already shipped (mode1 raw / mode2 final / mode3 clean-stream). Worth weighing what's left to gain.

One thing to un-conflate

tmux and CLI streaming are the same CLI, same subscription — the only difference is the invocation (a live terminal vs. --print/JSON output). There's no pricing gap between them. The pricing gap is CLI (either mode) vs. direct API. See §Pricing below — this one is easy to conflate and worth keeping straight.

Four layers, four owners

Every other section on this page — transports, bridge scope, P0 contracts, the test policy — is a decision made inside one of these four layers. Get the layers straight first and the rest mostly falls out of "which layer does this belong to."

Layer 0 · third party

The CLI binaries

claude, codex, cursor-agent, pi. Not our code — ground truth for what actually happens.

Owns: real model behavior, tool execution, auth/session state, every undocumented quirk.

We own here: nothing. Observation and steering only.

Layer 1 · adapter

multi-llm-provider-go

Make each individual provider work correctly on its own terms. No cross-provider abstraction or masking — that's Layer 2's job.

Owns: flags/env/launch, tmux vs. JSON parsing, shutdown, every per-provider quirk — plus the P0 contract tests proving it against real binaries.

Does not own: deciding what a session's system prompt/tools/config should be, or presenting any unified cross-provider interface.

Layer 2 · library

mcpagent

Mirror image of Layer 1's job: mask which provider and transport is underneath, one consistent interface on top. A caller shouldn't need to know or choose Claude-over-tmux vs. Codex-over-JSON.

Owns (the mandate): NewAgent + options, system prompt (incl. resume), MCP bridge, skills, multi-turn state, steer-vs-query — plus masking which provider/transport handles a turn, conversation persistence, and cost accounting.

Built vs. mandate: transport still an explicit per-provider flag, not auto-masked. Conversation persistence + cost accounting are now built (convrecord — see §Backlog) alongside system-prompt/bridge/skills/multi-turn/steering.

Layer 3 · application

family-server, AgentWorks, sparkquill…

The actual product. Depends on Layer 2 — free to route around it.

Owns: which UI a surface gets, which library mechanisms it actually calls, workflow/scheduling, data persistence — and building its own parallel mechanism when the library's doesn't fit.

Does not own: CLI process mechanics or Agent-construction internals, though it can bypass them.

Where bugs actually hide — and the P0/P1 numbers per layer

A layer's tests can only see as far as the layers below it that it actually calls into. A layer bypassing the one below — Layer 3 rolling its own skills delivery instead of Layer 2's WithAttachedSkills — creates a seam no lower-layer test can cross. The three lanes

Not a binary choice

Each is a genuinely different shape, not a setting on the same thing.

Lane one

tmux — the CLI, driven like a person would

A real terminal session stays alive across turns. Messages go in as keystrokes; the reply is whatever the CLI prints. Everything the CLI's own interactive UI does — prompts, banners, approval nudges — comes along too.

Lane two

CLI structured streaming — the same CLI, headless

The CLI's own --print / JSON-output mode: no TUI, typed events for text and tool calls instead of ANSI to parse. Still the CLI, still the subscription — but (today, in most providers) a fresh process per call, not a standing session.

Lane three

Direct API — no CLI at all

Raw calls to the model provider's API, with your own tool-calling loop. No subscription pricing, no CLI process, no terminal, no local session state — a plain metered HTTP request.

Worth unfolding

The dimensions that need more than one row

Steering — the one capability streaming can't match
Streaming / API con Not yet in your framing

Steering — a follow-up landing mid-turn instead of queueing — needs a process still alive to receive it. tmux has that process; a one-shot streaming or API call doesn't, so there's nothing to inject into.

Both AgentWorks (live-input mid-workflow) and sparkquill (mid-turn steer) depend on this today. Moving either to streaming or API means giving up steering — no replacement exists for a stateless channel.

Extraction clarity — how much of "streaming's advantage" is already won
Partially already solved

Raw tmux output is messy — ANSI, banners, tool calls interleaved with prose. But the transcript/marker parsing already shipped produces three clean views from that same session: raw terminal, final plain-text answer, and a clean streaming view with tool calls as their own event type.

So "streaming is smooth, tmux is messy" is true of raw tmux, less true of tmux-plus-parsing. What streaming adds on top: no ANSI to defend against, and typed events independent of transcript format. Real, but narrower than "solves an unsolved problem."

TUI noise — approval prompts, model-switch nudges
tmux con — confirmed Relocates, doesn't vanish, in streaming

Real and ongoing: Codex nudging to switch models, Cursor's approval flow — per-provider TUI behavior the CLI author controls. Flags suppress most of it, but every CLI update can introduce a new one — a maintenance tail, not a one-time fix.

Headless/streaming mostly removes this category, since there's no TUI to prompt in. It doesn't remove provider quirks generally — the pain relocates from "TUI prompts" to "schema/event edge cases," it doesn't disappear.

Is tmux actually cheaper than CLI streaming?
No — this was conflated

tmux and CLI structured streaming are two ways of invoking the same CLI under the same subscription — no price difference between them. The real pricing distinction is CLI (either mode) vs. direct API: flat subscription vs. metered per-token, no ceiling.

One historical exception: Claude's -p mode briefly had different pricing from interactive — since removed, no longer applies.

Less-confident nuance: "resume every turn" could cost more if it resent the full transcript as tokens, but native --resume reloads from the CLI's own on-disk session store instead — so the likelier cost is latency/process-startup, not a token bill difference (probable, not confirmed against vendor billing docs).

Net: tmux's real case over streaming is multi-turn feel and steering, not price. Price only distinguishes CLI from direct API.

Multi-turn resume cost
Streaming / API con — confirmed, and larger than it first looks

tmux's warm-session model treats "reload full context from scratch" as a rare fallback — only on session loss. A one-shot streaming/API channel has no warm session to fall back from, so it pays that cost on every single turn. tmux's fallback path is streaming's normal path.

Debugging: tool timing, transcripts, token usage & pricing
tmux already delivers this — proven tonight One nuance on what "pricing" even means here

Tool-call timing

All three can get this; the difference is effort. tmux derives it from transcript timestamps — real, proven, but a reconstruction whose quality varies by provider (Claude solid, Pi's marker-only streaming coarser). CLI streaming carries explicit timestamps as a first-class field. Direct API gives full control — you time it yourself.

Forensic transcripts

tmux's raw terminal capture is the richest record — literally what a human would have seen. That doesn't carry to streaming/API — no screen to have shown anything. What does carry: the CLI's own on-disk session transcript exists under tmux and CLI streaming equally. Streaming keeps the structured record without the raw-screen value; tmux keeps both.

Token usage & pricing

Already works under tmux — confirmed live for Claude Code: input_tokens=10060 output_tokens=110 cache_tokens=10050, total_cost_usd=0.0013, pulled from the CLI's own transcript metadata. CLI streaming's typed usage event is likely at least as clean. Direct API's usage field is most authoritative — no parsing at all.

The nuance — what the number means differs by lane

Under a flat-rate subscription (tmux or streaming), that cost figure isn't a real bill — it's an internal accounting estimate from published per-token rates, useful for attribution but not literal money changing hands. Under direct API, the same-shaped number is the real bill.

Reframing a claim

"Doesn't run on a server" — not quite the right drawback

tmux-driven CLIs already run on headless servers, all night. What's real is narrower: an interactive tool built for a person at a keyboard needs real engineering to behave reliably with no person present — that's where the cost sits.

  • Approval / sandbox flags — every provider has its own way to suppress interactive prompts.
  • Session lifecycle — keep-alive registries, idle-reaping, crash/resume detection.
  • Initial auth — one interactive browser login; reusable after.
  • Directory concurrency — tmux/CLI-state specific, not a streaming or API concern.

Direct API sidesteps all four structurally. That's the real shape of "harder to automate": not unavailability, an ongoing engineering tax.

Resolved — not tmux-specific

Why plan steps run in isolated folders

AgentWorks runs each plan step in its own temp folder because multiple CLI sessions can't run concurrently in the same working directory. Resolved — cause per docs/WORKFLOW_STEP_ISOLATION.md:

The actual mechanism

The orchestrator writes fixed-name instruction files (CLAUDE.md, AGENTS.md, …) into the working directory before every call. Two concurrent steps against the same folder race on those files — nothing to do with tmux mechanics; file-projection concurrency at the orchestrator level, and streaming hits it identically since it needs the same files on disk.

Fix already shipped (WithIsolatedSessionWorkspace, transport-agnostic). Direct API is exempt for a different reason — no local CLI reading instruction files at all.

The product angle

What users actually expect to see

The transport question keeps getting settled by a UI decision, not the other way around: most users don't want to see a terminal. A raw pane reads as confusing and unprofessional to a non-developer. Clean, formatted assistant messages are the default expectation.

General users — the default

Parents, kids, anyone not writing code for a living. A terminal in the middle of a chat reads as broken, not powerful.

Developers / advanced users — the exception

Want the raw terminal for transparency and debugging. AgentWorks' builder chat is built for this audience specifically — not the general case.

Not free either way

Showing a live terminal is its own engineering subsystem — emulator, its own transport, ongoing maintenance — independent of backend transport chosen. A clean-text UI skips all of it.

Since a clean-text UI never renders the terminal, tmux-vs-streaming for that surface was never about what the user sees — mode1/2/3 extraction already produces clean text from tmux fine. It's the terminal-view requirement that makes tmux non-negotiable for builder chat, not tmux's other properties.

A separate axis — routing, not transport

Does everything need to go through the bridge?

The MCP bridge's original justification was security through file/folder confinement. That doesn't actually hold: execute_shell_command runs outside any CLI's sandbox specifically so it can reach the real workflow directory, and supporting arbitrary user-installed CLIs (AWS CLI, gcloud, kubectl…) makes a folder-confined shell tool unworkable regardless. "The bridge keeps things confined" was never quite true.

Not in question

This does NOT mean drop the bridge. Custom app tools have no native equivalent at all — pure app logic, only possible bridge-shaped. The real question is narrower: for the two generic primitives — shell and file reading — does bridge-routing still earn its keep once confinement is off the table?

File reads — lean native

Low risk by nature — no mutation, nothing urgent to audit. Bridge-routing's advantages (audit trail, consistency) buy little here, while going native removes a real cost: the extra HTTP hop, and the cold-turn MCP-readiness race.

Shell execution — lean bridge-routed, for two specific reasons

Native shellBridge-routed shell
Audit trail Heuristic, lossy — order-only timing for Claude/Codex, char-count token estimate for Cursor (§Debugging) Real HTTP call into app-owned code — genuinely loggable, timeable, correlatable
Codex specifically Its functions.exec tool cannot be disabled by any flag — going native here doesn't weaken the "no native writes" guarantee, it deletes it entirely The only containment that exists for Codex at all — not a preference, a requirement
Latency / complexity Wins — no extra hop, no bridge-readiness race Real but bounded cost — same executor path already proven at scale tonight

Net: shell is the one action most worth having a reliable record of, and it's the one place Codex's unfixable native-exec problem is actually load-bearing. Reads don't carry either of those arguments.

Proposed shape

File reads go native, across all four providers. Shell stays bridge-routed as the default, non-negotiable for Codex. Native shell for the other three (Claude, Cursor, Pi — where native shell actually can be fully disabled, unlike Codex) is a separate, narrower call worth revisiting later, not a default to reach for now.

Ways to resolve it

Not binary — a few real shapes this could take

A

Stay tmux everywhere, invest further in the extraction layer

Keep one transport, one set of session/keep-alive machinery. Close the remaining gap with streaming (schema stability, less reliance on transcript formats) inside the tmux session rather than switching off it.

B

Split by the interactive/automated axis

Keep tmux for anything a human watches or might steer (both apps' chat surfaces, AgentWorks builder). Unattended plan steps could still move to streaming or API on their own merits (no steering need, no terminal to show) — just not because it removes the concurrency cost, since that cost is transport-agnostic (§Concurrency) and stays either way.

C

Investigate a persistent streaming loop, per provider

"Streaming means resume every turn" may not be universally true — some CLIs might support a standing process that accepts sequential prompts over stdin without a fresh cold start each time. Unverified today; worth checking per provider before assuming the resume cost is unavoidable.

D

Direct API for a future headless-only consumer

Don't retrofit it onto either current app. Reserve it for whatever eventually needs neither a terminal view nor steering nor subscription economics — a genuinely different shape of workload.

Live build log — real code, not just analysis

P0 contracts, for both transports

This system already has a formal "P0" concept for tmux (requiredP0CertificationIDs) — release-blocking guarantees. Structured/JSON transport had no equivalent bar, because it didn't functionally exist for text generation on 3 of 4 providers. Built out provider by provider, real e2e proof against installed CLIs, not stubs.

tmux — the existing P0 bar

From coding_agent_certification.go: "the contracts without which AgentWorks cannot safely run a coding CLI." Twelve, all release-blocking:

  • fresh_launch — starts in the right workspace
  • runtime_context — system prompt/skills/MCP runtime reach the provider
  • working_directory — provider operates in the exact caller workspace
  • trust_auth_prompts — startup trust/auth gates clear before the first prompt
  • mcp_bridge — bridge launches and tools are callable
  • slow_tool_false_idle — a long tool call is never misread as "done"
  • done_detection — the adapter knows when a turn is really finished
  • final_extraction — only the real answer is returned, never chrome/tool dumps
  • live_input / busy_live_input — steering works, including while busy
  • cancellation — context cancellation actually stops the process
  • parallel_isolation — concurrent sessions don't cross-talk

5 of 12 are about a live, ongoing process — categories that don't apply to a one-shot transport. This is why structured transport needs its own P0 bar, not a subset of this one.

Structured/JSON transport — the new P0 bar

Scoped to what a one-shot transport needs to be safe for its intended surface (unattended workflow steps) — a different shape, not a smaller tmux:

  • fresh_launch_structured — real CLI launch in structured mode, real text response
  • multi_turn_resume_structured — turn 2 recalls turn 1 via the CLI's own native session store, NOT a full-transcript resend: codex exec resume <id>, cursor --resume <id>, pi --session-id <id>, claude --resume <id>. Capture-on-fresh, resume-on-next; argv shape is unit-pinned per adapter (see §Test layer policy)
  • token_usage_structured — result/finish events produce non-zero, correctly-attributed token counts
  • tool_call_streaming_structured — tool calls stream as distinct start/end chunks, not buried in text
  • mcp_bridge_structured — a real MCP bridge tool is callable under this transport too
  • bridge_only_structured — built-in tools can be denied while bridge tools still work (the structured counterpart to tmux's hooks-based deny — see §Bridge scope)
  • error_handling_structured — a clear error surfaces on process failure or empty output, never a silent hang
  • graceful_shutdown_structured — the documented SIGTERM×3→SIGKILL sequence fires on the terminal event (docs/coding_sdk_structured_contract.md §9)

Not the same thing as an existing cert with a similar name

CertStructuredStreaming already exists and is already P0 — but it's clean text/tool-call extraction during a tmux turn, not about the --print/exec --json transport itself. Easy to conflate by name; genuinely two different things.

Codex: what building it live found

The design doc's event examples were wrong, not just stale — real shape (verified live): thread.started → turn.started → item.started/item.completed → turn.completed. Two real bugs found by running it:

  • MCP tool calls have their own item type. Adapter only recognized command_execution; mcp_tool_call was silently invisible — final answer still came through, just no tool-call chunks for bridge calls.
  • Wrong MCP config JSON shape. Codex expects a flat map; Cursor's {"mcpServers": {...}} convention silently produced an empty command, quietly dropped rather than erroring. Root-caused by reproducing directly against the real CLI, bisecting the config piece by piece.

Both fixed and re-proven live — all three tests pass against the real CLI. Existing Codex suite unaffected.

Pi: what building it live found

Never had a structured implementation to recover — built fresh from a captured event trace. Basic launch and MCP bridge passed clean first try — the JSON-format lesson from Codex (Pi needs Cursor's wrapper shape, not Codex's flat map) paid off immediately.

Bridge-only took three attempts — each failure pointed at the next fix

1: --no-extensions strips ALL extensions, including Pi's own MCP mechanism — bridge vanished with native tools, no text at all.
2: --no-builtin-tools (the semantically correct flag, matching tmux) — native containment worked, but MCP also vanished: the flag turns out to suppress default extension auto-discovery too, undocumented.
3 (correct): --no-builtin-tools + explicit -e npm:pi-mcp-adapter re-add. All three tests now pass: native write blocked, bridge works, tool-call events stream correctly.

New policy — did not exist before tonight

Which e2e test lives at which layer

Checked both repos — no existing doc had this answer. Genuinely ad hoc, and it cost real debugging time: a system-prompt-clobbering bug (57b4dd9) shipped unnoticed by every adapter-level test, because it structurally couldn't be seen from that layer.

The rule

Test ownership follows where the logic lives. Real CLI process behavior (flags/env) → adapter-layer test (multi-llm-provider-go). How a Go function constructs the call before the adapter sees it → library-layer test (mcpagent). Both, not either/or: the adapter test is fast and narrow-blast-radius; the library test proves the actual path a real consumer exercises — the only place a bug like 57b4dd9 is visible at all.

Judging output quality: agent sign-off, not an external LLM judge

Final-response extraction quality ("is this the clean answer, or did tool chrome leak in?") is a semantic call a string assertion can't make. It used to be scored by a Gemini/Vertex judge — which meant a real test needed GEMINI_API_KEY to run at all, and failed hard without it. Now it routes through agentreview: the real extraction is recorded to a committed JSON, and the agent running the tests reads it against the extraction rubric and signs off on the current fingerprint. Same semantic judgment, no API key, no network, verdict committed to the repo — and because a fixture extraction is deterministic, one sign-off persists until the parser's output actually changes.

A fourth thing this exposed: checkouts drift

The app repo (agent_go/family-server) doesn't depend on the same local checkouts this doc's work happened in — separate clones entirely. Three+ local forks of the same libraries can silently diverge; "which checkout does this app build against" (grep replace go.mod) is worth checking before assuming a fix is reachable from a given app.
